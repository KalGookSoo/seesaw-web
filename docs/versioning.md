## 시맨틱 버저닝

시맨틱 버저닝은 소프트웨어의 버전을 체계적으로 관리하기 위한 규칙입니다. 이 규칙을 따르면 버전 번호만으로도 해당 릴리스의 성격을 파악할 수 있습니다.

## 버전 번호의 구성

버전 번호는 `주.부.수` (Major.Minor.Patch) 형식으로 구성됩니다.

### 주 버전 (Major)
- 이전 버전과 호환되지 않는 변경사항이 있을 때 증가
- 예: API 구조 변경, 데이터베이스 스키마 변경
- `1.0.0 → 2.0.0`

### 부 버전 (Minor)
- 이전 버전과 호환되는 새로운 기능 추가 시 증가
- 예: 새로운 API 엔드포인트 추가
- `1.0.0 → 1.1.0`

### 수 버전 (Patch)
- 버그 수정이나 성능 개선 시 증가
- 예: 오타 수정, 성능 최적화
- `1.0.0 → 1.0.1`

## 버전 식별자

### SNAPSHOT
- 개발 진행 중인 버전
- 예: `1.0.0-SNAPSHOT`
- 매일 여러 번 빌드될 수 있음
- 항상 최신 개발 버전을 의미

### Alpha
- 초기 테스트 단계
- 주요 버그 존재 가능
- 예: `1.0.0-alpha`, `1.0.0-alpha.1`

### Beta
- 기능은 완성되었으나 안정화 필요
- 테스트 진행 중
- 예: `1.0.0-beta`, `1.0.0-beta.2`

### RC (Release Candidate)
- 출시 후보 버전
- 최종 테스트 진행 중
- 예: `1.0.0-RC1`, `1.0.0-RC2`

## 실제 사용 예시

### 개발 주기 예시
```
1.0.0-SNAPSHOT  # 개발 시작
1.0.0-alpha     # 초기 테스트
1.0.0-beta      # 기능 테스트
1.0.0-RC1       # 출시 후보 1
1.0.0           # 정식 출시
1.0.1-SNAPSHOT  # 버그 수정 개발
1.0.1           # 패치 릴리스
1.1.0-SNAPSHOT  # 새 기능 개발
```


### Gradle 빌드 예시
```shell script
# 개발 버전 빌드
./gradlew bootJar -PappVersion=1.0.0-SNAPSHOT

# 정식 버전 빌드
./gradlew bootJar -PappVersion=1.0.0

# 테스트 버전 빌드
./gradlew bootJar -PappVersion=1.0.0-beta.1
```


## 버전 관리 모범 사례

### 1. 버전 증가 규칙
- 주 버전: API 변경, 호환성 깨짐
- 부 버전: 새 기능, 이전 버전 호환
- 수 버전: 버그 수정, 성능 개선

### 2. 개발 단계별 식별자 사용
1. 초기 개발: `SNAPSHOT`
2. 알파 테스트: `alpha`
3. 베타 테스트: `beta`
4. 출시 준비: `RC`
5. 정식 출시: 식별자 없음

### 3. 버전 관리 장점
- ✅ 프로젝트 진행 상황 파악 용이
- ✅ 의존성 관리 효율적
- ✅ 배포/롤백 관리 체계적
- ✅ 팀 커뮤니케이션 명확

### 4. 주의사항
- 한 번 출시된 버전은 절대 수정하지 않음
- 모든 변경사항은 새로운 버전으로 관리
- 정식 출시 전 충분한 테스트 필요
- 변경사항 문서화 필수
